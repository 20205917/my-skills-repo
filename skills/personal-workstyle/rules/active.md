# Active Rules

- [R001] 默认使用中文交流与文档说明（除非用户明确要求其他语言）。
- [R002] 对于较大实现计划，必须先给 Draft，等待用户明确确认后才能固化到项目文档或执行。
- [R003] 每次阶段性交付输出采用固定结构：Done / Decision / Next。
- [R004] 若规则与项目现有文档冲突，先指出冲突并请求用户裁决，再更新规则。

- [R005] 不确定的事实必须明确标注为“假设”，不能当作结论输出。
- [R006] 改动前先明确验收标准（功能、测试、构建、发布门槛）。
- [R007] 复杂逻辑补充“为什么”的注释，不写“做了什么”的废话注释。
- [R008] 禁止大规模无意义格式化，避免噪音 diff。
- [R009] 一个问题一个 commit，提交必须可追溯。
- [R010] 提交前强制检查 git status，只提交目标文件。
- [R011] 中文规则同时作用于文档和 Git 提交（commit message 与 commit body）。
- [R012] 提交说明至少包含：根因、改动、验证、影响。
- [R013] 进度同步要短、频繁、可执行（在做什么、发现什么、下一步）。
- [R014] 最终输出固定结构：结果、改动、验证、风险、下一步。
